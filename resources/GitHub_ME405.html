Using Git in ME405

Git is the most popular Source Control Management (SCM) program at this time. It has approximately eleventy zillion useful features, of which we will use about eight in ME405.

The diagram above shows an overview of how files are exchanged between multiple teammates’ computers and a cloud repository such as GitHub™.

  • The files you’re actively editing are in a working directory on your PC.

  • You should edit files in the working directory, then run them using Thonny and copy them to the MicroPython board as needed. This is done as regular copying of files. It is not a good idea to edit files directly on the MicroPython board.

  • The local repository is managed by a Git program and contains a set of “snapshots” of your software at various points in time. You stage and commit changes periodically; this makes a new snapshot of important files in your working directory and saves that snapshot in the local repository.

  • When appropriate, you push snapshots from your local repository to the repository server in The Cloud.

  • When changes have been made to your project on another computer such as your teammate’s, you should pull those changes into your local repository and working directory. If you’ve both made changes at the same time, you may need to merge your changes.

If you have already installed Git tools for another course, you may continue to use those. However, giving the recommended ME405 tools a try is a good idea.

Install and Configure Git

If you’ve already installed Git, you can probably skip this section.

  • Go to https://git-scm.com/download and install the correct version for your OS.

  • Open a terminal (or PowerShell, or whatever your system calls it) and type the command

    git config --global user.name "username"

    where username is whatever name you have chosen to identify yourself (it does not have to be your actual name).

    You may want to set your Git email address at this time, but you may also want to not set your email until you get a special address from GitHub which won’t get you spammed.

    git config --global user.email "name@server.net"

Set up Your GitHub Account

  • Go to https://github.com and set up a GitHub account.

    Note: You are not required nor requested to use your actual name for the account, nor to make any personally identifying information visible in your GitHub repository.

Install a Git Client

If you’re already using SourceTree and actually like it, you may continue to use that. Otherwise, GitHub Desktop is a relatively easy to use Git graphical interface which is recommended. If you go on to professional work with large projects, a more fully featured client will be better, but for learning Git with small mechatronics projects, this client is a good choice.

  • Go to https://desktop.github.com/ and download a version for your operating system.

  • Although Linux isn’t listed by GitHub, there’s a script for Ubuntu at https://gist.github.com/berkorbay/6feda478a00b0432d13f1fc0a50467f1 which allows the program to be installed on a Debian-based Linux box. More information is at https://github.com/shiftkey/desktop.

Add Local Repository to your GitHub Account

It is assumed that you have already created a local Git repository – that is, a repository for your project on your computer – and made some commits to it. The following instructions are for GitHub Desktop:

  • Choose File -> Add local repository... and choose the folder in which your project files are. It’s best to choose the top-level project directory, the one containing your src and docs directories.

  • Add and commit any files which should be pushed to GitHub. It’s most important to add src/*.py and all files in docs.

  • Choose Repository -> Repository settings...

  • Check the Git config to verify that you’re using the name and email which you prefer for your GitHub account.

  • Go back to Remote and click the Publish button.

  • Change the name if desired. Add a meaningful description. Uncheck the Keep this code private button unless you need to restrict access (you’ll have to specifically authorize your instructor to access your repository if it’s private).

  • Click the Publish Repository button and authenticate if necessary. If you use the same Internet provider I do, this may take a few tries.

  • If you need to change GitHub repository settings, it’s most easily done using the Settings tab on the repository web page.

  • This might be a good time to add a README.md page for your repository. Create it in the top-level directory, edit it on your local computer, add it in GitHub Desktop, commit the change(s), and use Repository -> Push to push the changes to the GitHub remote repository. README.md is written in Markdown syntax, which is really easy to learn:

    https://docs.github.com/en/github/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax

Set Up GitHub Web Pages

This is a streamlined way to get your Doxygen generated documentation onto the Web. Each time you run Doxygen, then stage and commit and push changes, your documentation will be updated along with your source code.

If you’ve made a portfolio in ME305, this is a bit different: You’ll have one repository with its own set of documentation for each lab exercise, not a big portfolio with all your documents.

  • Make sure you have configured Doxygen according to the instructions in the Setting Up Doxygen page on Canvas.

  • Run Doxygen and make sure your folder called docs is filled with many files.

  • Stage (add) the entire /docs folder full of files and commit that change. Most Git GUI programs will stage the changed files automatically. You can commit document changes at the same time as code changes.

  • Push your changes to GitHub.

  • Go to your project’s main site on GitHub (you should see yourname/ProjectName at the top). Click the Settings tab at upper right. Choose the Pages category at left. Set the source branch to master and select the folder /docs. Save your choices.

    Important: It may take up to 20 minutes (according to the GitHub folks) before your documentation becomes visible in a Web browser, so don’t worry if you get some 404 errors for a while.

  • See also https://docs.github.com/en/pages.
